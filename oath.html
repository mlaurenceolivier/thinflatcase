Oath Publishing
"Engineers should be able to grab code, plug it in and not have to worry about being designers.”

Role: Lead Designer | When: 2018 | Tools: Sketch/Invision App, Terminal/Github, SCSS/SASS, HTML/MD

It was decided that we would merge several different teams from various parts of Aol into a publishing suite to service all of our O&O brands (HuffPo, Engadget, TechCrunch, Autoblog, etc.). As part of this reorg, we needed a cohesive branding with which to build all of these products on.

We determined the following from the get go:

  • Overall style (material design)
  • Logo (Large P) and Branding (Pink)
  • Foundation (Twitter Bootstrap)

From here, it was fairly straight forward on creating everything else (buttons, form fields, search bars, nav bars, etc.). Although we had consolidated several teams, we did not consolidate enough front-end designers and our engineers were not willing (or ready) to make the jump to designers. Even with 3 designers on our team of nearly 100, we couldn’t generate mocks fast enough to keep development of all of our suite of products moving.

SOLUTION

Build a living style guide as a GitHub repo that would allow engineers and developers to easily grab code snippets, paste it into their applications and have everything styled beautifully without them needing to write any CSS.

The basic repo (not updated) is still alive and well: athena-styleguide.aol.com

Throughout this process, I had the help of one other designer (with a background in front-end development) that helped to get the basics set up while I managed creating all of the elements and writing SCSS for the code snippets. Teamwork!

TEACHING

Lastly, we had a slew of documentation to manage and it was decided we would write everything in Markdown and contribute everything via Github. All of our Product Managers were going to contribute the documentation, the only problem: none of them knew how to contribute via Github.

With my ability to easily communicate with developers and designers, I took it upon myself to train each Product Manager in the basics of writing Markdown files, merging files, and making pull requests. Sure, there were several hiccups with conflicts and all, but overall, I was able to help explain Github in the simplest of terms to the PMs and get them up and running.
